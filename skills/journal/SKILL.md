# Journal — generate the AI daily journal

This skill is the orchestrator. It runs in the main conversation (where the `Agent` tool actually works for fanning out to `journal-analyzer` subagents — Claude Code subagents cannot themselves spawn subagents, so this work cannot live in a subagent).

The downstream consumer is the **resume-builder** subagent invoked by `/career:resume`, so the journal you emit must be:

- **Machine-parseable** — frontmatter carries the structured signal (activities, accomplishments, categories) that the resume-builder consumes.
- **Human-readable** — the body is for the user to skim and trust.
- **Honest about scope** — never inflate or invent; the resume-builder will quote your accomplishments verbatim.

## Accepted arguments

| Form                           | Meaning                                                     |
| ------------------------------ | ----------------------------------------------------------- |
| (none)                         | Today, in the user's local timezone                         |
| `today`                        | Today                                                       |
| `yesterday`                    | Yesterday (the recommended form for scheduled/cron runs)    |
| `YYYY-MM-DD`                   | That single date                                            |
| `YYYY-MM-DD..YYYY-MM-DD`       | Inclusive range; produces **one journal file per day**, processed serially |

## Configuration

Read `~/.career/config` (YAML) to determine where journals live and how to fan out. Expected fields:

```yaml
journals: ~/career/journals/   # required — directory where journal markdown files are written
# Optional:
# timezone: Asia/Taipei
# categoryLookbackDays: 30
# analyzerBatchSize: 5         # max analyzers run in parallel per wave (default 5, hard-capped at 10)
# maxAnalyzers: 30             # warn-and-confirm threshold for total project count per day (default 30)
```

- If `~/.career/config` is missing, fall back to `~/.career/journals/` and proceed (do not error — first-run should just work).
- Expand `~` to the user's home directory.
- If the journals directory doesn't exist, create it.
- The journals directory **may be a git repo** — if so, your output will become a tracked file. **Do not auto-commit, push, or alter git state** unless the user explicitly asks. Surface git status in the final summary instead.

## File ownership in the journals directory

Two file types coexist in `<journals_dir>/`:

| Filename             | Owner        | Behavior                                                                                |
| -------------------- | ------------ | --------------------------------------------------------------------------------------- |
| `YYYY-MM-DD.md`      | **AI-only**  | Written and regenerated by this skill. You may freely rewrite these files.              |
| `YYYY-MM-DD.user.md` | **User-only**| Authored by the user via `/career:user-journal`. **Never read or modify these files.**  |

The `.user.md` files carry the user's own reflections, corrections, and off-screen accomplishments. Downstream tools like the resume-builder may consume both files as a pair, but this skill's scope is strictly the `.md` side. This separation removes any risk of overwriting user-authored content — the AI file can be regenerated freely because no human edits live there.

When scanning the journals directory (e.g., for `knownCategories` in step B.1), **filter to files matching the pattern `YYYY-MM-DD.md` only** — exclude `*.user.md` and any other files.

## Procedure

### A. Parse the argument and enumerate dates to process

1. **Resolve the timezone first** using this chain (highest precedence first):
   1. `timezone:` field in `~/.career/config` (a YAML key, optional). Read with a `Bash` one-liner — `python3 -c "import yaml,os; print(yaml.safe_load(open(os.path.expanduser('~/.career/config'))).get('timezone',''))"` — and trim whitespace.
   2. The system's local timezone (e.g., `$TZ` or whatever Python's `datetime.now().astimezone()` resolves to).

2. **Resolve the argument into a list of calendar dates**, in chronological order:
   - (none) / `today` → `[today]` in resolved timezone.
   - `yesterday` → `[yesterday]` in resolved timezone.
   - `YYYY-MM-DD` → `[that-date]`.
   - `YYYY-MM-DD..YYYY-MM-DD` → every inclusive date in resolved timezone.
   - Reject future dates, inverted ranges (`end < start`), and malformed input — ask the user to retry rather than guessing.

3. **Emit a run preview** before starting the loop, so the user knows what's about to happen. Example: `Generating journals for 34 day(s): 2026-04-02 → 2026-05-05.` For a single date, say `Generating journal for 2026-05-05.` Keep it to one line.

4. **For each enumerated date**, run the per-day procedure (section B) **in chronological order, serially**. Serial ordering matters: day N's `knownCategories` includes slugs minted on days 1..N-1, which is how soft-consistency converges. Parallelizing days would force every day to mint independently and produce drift.

5. **Continue past failures.** If one day's procedure errors, record it in the cross-day summary and move on to the next day. A bad day must not block the rest of the range.

6. After all days are processed, emit the cross-day summary (section C).

### Progress reporting

The skill must keep the user informed without spamming them. Use these checkpoints:

- **At the start of each day** (before step B.1), emit one short line: `[i/N] 2026-04-02 — discovering projects…` where `i` is the 1-based day index in the run and `N` is the total. For a single-date run, you may omit `[i/N]`.
- **After step B.2** (project discovery), emit one line summarizing the count: `[i/N] 2026-04-02 — 4 projects in range, 2 waves of 2.` If zero projects, say `0 projects in range; nothing to write.` and skip to the per-day status.
- **After each wave** (within step B.3), emit one line: `[i/N] 2026-04-02 — wave 1/2 done (4 ok, 0 skipped).` This is the longest-running step, so a heartbeat per wave is what keeps a multi-project run feeling responsive.
- **At the end of each day** (after step B.9), emit the per-day status (section B.10) immediately, not held until the end. One short block per day, not a paragraph.
- **At the end of the whole run**, emit the cross-day summary (section C).

Every progress line is plain text in the conversation — these are user-facing messages between tool calls, not tool output. Keep them under one line each. Do **not** dump raw analyzer JSON, full file contents, or per-accomplishment detail into progress lines; those belong in the journal file.

### B. Per-day procedure

Computed inputs for this day:
- `targetDate` — the calendar date being processed.
- `start` / `end` — `[YYYY-MM-DDT00:00:00<resolved-offset>, YYYY-MM-DDT23:59:59<resolved-offset>]`, then converted to **UTC** ISO-8601 strings before passing to analyzers.

#### B.1. Load known categories (soft-consistency pre-pass)

- Look at the journals directory. **Filter to files matching `YYYY-MM-DD.md` only** — skip `.user.md` and anything else. By default scan the most recent **30** matching files (sorted by date desc).
- For each, extract the `categories:` list from the YAML frontmatter — a `Bash` one-liner with `python3 -c` is enough; do **not** read the full files.
- **If a journal file for `targetDate` already exists** (re-run for the same day), include its `categories` list as well. Otherwise the second run of the day could re-mint slugs the first run already established.
- **For multi-day ranges**, also include any `categories` already aggregated on prior days **of this run**. Pass the union forward so day 3 benefits from day 1 and day 2.
- Union and dedupe into `knownCategories`. This list is passed to every analyzer in step B.3.
- If the journals directory is empty (first run), `knownCategories` is `[]`.

#### B.2. Discover candidate projects

- List `~/.claude/projects/*/` with `Glob`.
- For each project directory, do a fast `Bash` check: does any `.jsonl` contain a `user` or `assistant` record whose `timestamp` falls in `[start, end]`? Use a streaming Python one-liner — do **not** `Read` the JSONL files into context.
- **This existence check is the only permitted JSONL touch in this skill.** It returns a boolean per project — nothing else. Do not extract message text, file paths, commit hashes, categories, accomplishments, or any other content from the JSONL here or elsewhere in the skill, whether via `Bash`, `Read`, `Grep`, `Glob`, or any other tool. All content extraction is delegated to `journal-analyzer` in step B.3 (see Constraints).
- Keep the directories that pass the check.
- **Sort the surviving list by encoded directory name, ascending.** Stable sort means the same project set always produces the same wave assignments, which keeps cross-wave category learning deterministic across runs.

#### B.3. Fan out to journal-analyzer (batched, wave-based)

**Hard precondition: every project that survives step B.2 must be analyzed by a `journal-analyzer` subagent invoked via the `Agent` tool — no exceptions.** There is no inline-extraction fallback. If you find yourself writing a `Bash`/`python3` script to pull message text, file paths, commits, or categories out of JSONL, **stop** — that work belongs inside `journal-analyzer`, which has its own context window and is the only component allowed to author enrichment fields (`starComponents`, `qualityTier`, `scope`, `context`, `impact`). Bypassing it for any reason — context budget, turn budget, "it's faster", or otherwise — produces a journal that fails its contract with the downstream `resume-builder`. If the project count is too high to delegate, raise it via the `maxAnalyzers` warn-and-confirm gate below; do not silently substitute inline parsing.

**If the `Agent` tool itself errors out** (e.g., the runtime can't dispatch the subagent for some reason), **abort this day's run and write nothing for `targetDate`.** Record the failure in the per-day status. Do not fall back to inline JSONL parsing under any circumstance. Move on to the next day.

Naive fan-out doesn't scale. With N projects, holding N raw analyzer responses in context while you compose the final journal can exhaust the window, and the cost scales linearly. Apply batch control:

##### Pre-flight: check the project count against budget

Let `N` = number of candidate projects from step B.2, and `maxAnalyzers` = the config value (default 30).

- If `N <= maxAnalyzers`: proceed.
- If `N > maxAnalyzers`: **do not silently fan out.** Report:
  - The count
  - The first 10 project names so the user can see what's in scope
  - Three suggestions: narrow the time range, pass a project filter, or raise `maxAnalyzers` in the config
- Wait for confirmation before proceeding. In a non-interactive context (e.g., scheduled run), fail this day with a clear error rather than running a runaway job.

##### Wave execution

Let `batchSize` = `min(analyzerBatchSize from config, 10)`. Default `batchSize = 5`. The hard cap of 10 protects the main conversation's context regardless of config.

Split the `N` projects into waves of `batchSize` each. For each wave (in order):

1. **Recompute the wave's `knownCategories` input.** Take the original `knownCategories` from step B.1 and union it with every category slug currently present in the running aggregate (i.e., contributed by prior waves of *this day's* run, post-normalization). Pass this enriched list to every analyzer in the wave. **This recompute happens before each wave is dispatched** — wave 1 sees only B.1's list; wave 2 sees B.1 + wave 1's slugs; wave 3 sees B.1 + waves 1-2's slugs; and so on. This is what gives later waves the chance to reuse slugs earlier waves established, tightening drift within a single day.
2. **Issue the whole wave in one message** containing `batchSize` parallel `Agent` tool calls. This is where the parallelism speedup comes from.
3. Each `Agent` call passes:
   - The encoded directory name (and the canonical path if you have it)
   - The same time range (`start`, `end`)
   - The wave's enriched `knownCategories` from step 1
   - The output schema you want back (default: the one in `journal-analyzer`'s spec)
4. **When the wave returns, run B.4 immediately** (parse → skip empty → record failures → normalize categories → fold → discard). Do not hold raw analyzer outputs across waves.
5. **Failure isolation within a wave:** if one analyzer errors or returns malformed JSON, record `{ project, reason }` in `skippedProjects` and keep processing the rest of the wave. A bad analyzer must not block a good one.
6. Proceed to the next wave.

##### Analyzer time budget

The analyzer has `maxTurns: 20` set in its frontmatter. If it exhausts that budget (pathological JSONL, too many sessions), the runtime returns early. Treat this as a skip with reason `exceeded turn budget` — do **not** retry in a loop.

#### B.4. Collect, normalize categories, and aggregate (per wave)

Steps B.3 and B.4 interleave — each wave's outputs are normalized and folded into the running aggregate before the next wave is issued, then the raw responses are dropped from context. The aggregate itself is what survives between waves.

**Order matters.** For each wave's analyzer outputs, do these steps **in this exact order** before moving to the next wave:

1. **Parse** each JSON response.
2. **Skip empty:** projects that returned `sessionsCount: 0` are valid but contribute nothing — drop them quietly.
3. **Record failures** in `skippedProjects` with `{ project, reason }` (malformed JSON, runtime error, etc.).
4. **Normalize minted categories** (the substance of step B.5 below) — operate on the **still-in-hand** raw outputs of this wave, **before** folding. The substitution targets (per-accomplishment `categories[]`, per-session `categories[]`, `projectSummary.categories[]`, `projectSummary.newCategories[]`) only exist in the raw outputs; once folded, the per-output shape is gone.
5. **Fold into the running aggregate** (see step B.6's schema for the target shape): append new accomplishments, union activities/categories, accumulate per-project entries, collect any `newCategories` slugs that survived normalization.
6. **Discard the raw responses** once folded. This is the core trick that makes large N tractable.

By the time the last wave completes, you should have a single consolidated aggregate object and a `skippedProjects` list — nothing else from the analyzers should remain in context.

#### B.5. Category normalization rules (referenced from B.4 step 4)

For each minted slug across this wave's analyzer outputs, decide:

- **Approve** — no existing slug in `knownCategories` (or in other analyzers' outputs from this run) is a reasonable substitute. Keep the new slug; it will flow into the aggregate's `newCategories` in B.4 step 5.
- **Replace** — an existing slug fits well enough. Substitute the existing slug for the minted one **everywhere it appears in that analyzer's raw output**: each accomplishment's `categories[]`, the session-level `categories[]`, the `projectSummary.categories[]`, and remove it from `projectSummary.newCategories[]`. After this substitution, the fold step (B.4 step 5) will see only the canonical slug.

How to judge a "reasonable substitute":

- **Synonyms / alternate spellings** — `nextjs`, `next-js`, `next.js` → use whichever form is already in `knownCategories`.
- **Sub-topic absorbed by a broader existing slug** — minted `react-hooks` while `react` exists → replace with `react` (specificity is a goal but consistency wins on the first run; the user can split later if needed).
- **Same domain, different framing** — minted `auth-flow` while `auth` exists → replace with `auth`.
- **Acronym vs. expansion** — minted `json-web-tokens` while `jwt` exists → replace with `jwt` (the existing form wins).
- **Do not force a replacement** when the topics are genuinely distinct: `payments` and `billing` are not the same; `react-native` and `react` are not the same.

Track every replacement so the status block can report it: `{ from: "react-hooks", to: "react", project: "<name>" }`. After this step, only **truly new** slugs remain in `newCategories` for the user to review.

#### B.6. Final aggregate shape (target for B.4 step 5)

After the last wave's fold, the running aggregate from B.4 should hold:

- **`activities`** — union of all activity slugs across projects (deduped).
- **`accomplishments`** — flattened list of every accomplishment from every project, each annotated with its source `project` name. Preserve **all** analyzer-provided accomplishment fields verbatim: `text`, `categories`, `evidence`, and the enrichment fields (`qualityTier`, `techStack`, `scope`, `context`, `impact`, `starComponents`) when present. Do **not** rewrite, remove, or "improve" any of these — the analyzer authored them for the downstream `resume-builder`.
- **`techStack`** — union of all per-accomplishment `techStack` arrays across all projects (deduped, sorted alphabetically). This gives the resume-builder a fast scan of technologies used without parsing every accomplishment.
- **`categories`** — union across all projects (deduped, sorted alphabetically for stability).
- **`activityByProject`** — `{ project_name: [activity_slug, ...] }` for the body section.
- **`newCategories`** — union of every analyzer's `projectSummary.newCategories` *that survived B.5 normalization* (replacements have been substituted out by then). These are the slugs actually minted in this run that did not exist in `knownCategories`. Surface them prominently so the user can review for drift.
- **Counts** — `projectsCount`, `sessionsCount`, `accomplishmentsCount`.

#### B.7. Synthesize cross-project narrative

The skill's value-add is **cross-project insight** — the analyzer already covered per-project detail. Write a short narrative that covers:

- **Headline** (1-2 sentences) — the single most resume-worthy outcome of the day, across everything.
- **Themes that span projects** (e.g., "auth refactoring across two repos", "debugging-heavy day, mostly observability gaps").
- **Time allocation hints** — which project absorbed the most sessions / accomplishments.
- **Context-switch count** — number of distinct projects.
- **Open threads** carried across projects.

Do not invent accomplishments here. Themes summarize patterns; concrete bullets stay in the `accomplishments` list.

#### B.8. Emit the journal document

Single Markdown document with YAML frontmatter. Frontmatter is the **machine-readable contract** for the resume-builder; the body is for humans.

```markdown
---
schemaVersion: 1
date: <YYYY-MM-DD>          # targetDate
range:
  start: <ISO-8601 UTC>      # the day's UTC start
  end: <ISO-8601 UTC>        # the day's UTC end
projectsCount: <N>
sessionsCount: <M>
accomplishmentsCount: <K>
analyzersInvoked: <count of journal-analyzer Agent calls issued for this day>
activities: [development, debugging, ...]
categories: [auth, jwt, react, ...]
techStack: [react, typescript, jwt, postgres, ...]
newCategories: [jwt]
projects:
  - name: <project-name>
    cwd: <canonical path>
    sessionsCount: <n>
    activities: [...]
    categories: [...]
accomplishments:
  - text: "Resume-bullet-style sentence (verbatim from analyzer)"
    project: <project-name>
    categories: [react, auth]
    evidence:
      filesTouched: [src/auth/middleware.ts]
      commands: [pnpm test auth]
      commits: [a1b2c3d]
    qualityTier: high
    techStack: [react, typescript, jwt]
    scope: { filesChanged: 4, servicesAffected: 1 }
    context: { situation: "...", challenge: "..." }
    impact: { metric: "...", type: reliability }
    starComponents: { situation: "...", task: "...", action: "...", result: "..." }
---

# Journal — <YYYY-MM-DD>

## Headline
<1-2 sentences: the most resume-worthy outcome of the day>

## Activities
- **Development** — <project-a>, <project-b>
- **Debugging** — <project-a>
- **Refactoring** — <project-c>
<only list activities that actually occurred; map each to the projects it appeared in>

## Accomplishments
### <project-a>
- `[react, auth]` Implemented JWT refresh-token rotation in `auth/middleware.ts`, eliminating silent session expiry on the dashboard.
- `[testing]` Added integration tests covering the refresh path against a real Postgres fixture.

### <project-b>
- `[ci-cd]` Migrated CI from CircleCI to GitHub Actions, reducing pipeline cost by removing the unused parallelism tier.

## Themes
- <Cross-project pattern 1>
- <Cross-project pattern 2>

## Open threads
- <project-name>: <unfinished work or follow-up>

## Categories touched
<comma-separated list of all category slugs, alphabetical>

## New categories this run
<list any slugs from `newCategories`, with the project they came from. If none, omit the section.>
```

#### B.9. Write and merge (idempotent)

Output path: `<journals_dir>/<targetDate>.md`.

**If no file exists at the path:** write a fresh document using the schema in step B.8.

**If a file already exists** (re-run, or earlier run that hit a different subset of sessions), perform an idempotent merge:

1. **Read** the existing file. Parse the YAML frontmatter and locate the existing `accomplishments`, `activities`, `categories`, `projects`, and `newCategories` lists.
2. **Merge accomplishments — dedupe by `(project, text)` exact match.**
   - Same project + identical accomplishment text → keep only the existing entry; do **not** create a duplicate.
   - If the new entry has stricter evidence (e.g., a commit SHA the old entry lacks) → enrich the existing entry's `evidence` arrays (union, deduped) but leave `text` untouched.
   - **Enrichment-field merge (deterministic completeness heuristic).** For each enrichment field, score existing vs. new on objective signals; the higher score wins. **Ties go to the existing version** so re-runs are stable when nothing actually improves. Per-field rules:
     - `techStack` — union of both arrays. A strict superset is always at least as good.
     - `scope` — take the max of each numeric sub-field (`filesChanged`, `servicesAffected`, `endpointsBuilt`). These counts are monotonic over a re-extracted session.
     - `qualityTier` — **existing always wins.** It's a label, not data; flapping between `medium`/`high` across re-runs is worse than stale.
     - `starComponents` — score = count of populated sub-fields among `situation`/`task`/`action`/`result`. The version with more components wins; pick per-field from each side if you want the union (only when both versions are non-empty for a given field, prefer existing).
     - `impact` — score: 2 if `metric` contains digits, percent signs, or units (e.g., "ms", "s", "x"); 1 if `metric` is present without numbers; 0 if absent. Numbers are the whole point of `impact`.
     - `context` — score: length within 30–500 chars wins; outside that range scores 0 (catches both empty stubs and runaway hallucinations).
     - For any field the existing entry lacks entirely, the new value wins by default (no comparison needed).
   - New project + new text → append.
3. **Merge activities, categories, and techStack** — set union, deduped. `techStack` is sorted alphabetically for stability.
4. **Merge `newCategories`** — keep only slugs that are *actually* new at merge time. Exclude any slug already in the existing file's `categories` list (whether placed there by an earlier run for the same date *or* by an earlier day in this same range — both surface via `knownCategories` in step B.1).
5. **Merge `projects`** — by `name`. For an existing project, union its `activities` and `categories`, sum its `sessionsCount`. For a new project, append.
6. **Recompute counts** — `projectsCount`, `sessionsCount`, `accomplishmentsCount` from the merged data, not by adding to old values (sums drift if the same session is reanalyzed). Recompute `analyzersInvoked` as the count of analyzer calls issued in this run only.
7. **Rewrite the body** to reflect the merged frontmatter — accomplishments grouped by project, activities mapped to their projects, themes regenerated from the full merged set. Do **not** try to surgically patch the old body; regenerate it from the merged frontmatter to keep them in sync. Because `.md` is AI-owned, full-body regeneration is safe — user-authored content lives only in `.user.md`.
8. **Write** the merged document with the `Write` tool.

**Important:** the merge is purely additive — never delete an accomplishment, project, or category that was in the prior file unless the user explicitly asks.

#### B.10. Per-day status

**Emit immediately** as a short user-facing block (the day's "done" message), and **also retain** the same record for the cross-day summary in section C. A status record for `targetDate` includes:

- Output file path and whether it was a fresh write, an idempotent merge, or **not written** (analyzer failure / preflight gate).
- Counts: projects analyzed, sessions covered, accomplishments emitted (and how many were new vs. deduped).
- **`analyzersInvoked`** — the literal count of `journal-analyzer` `Agent` calls issued in step B.3. This must equal `projectsCount + len(skippedProjects)` (every surviving project triggers exactly one analyzer call, succeed or fail). If `analyzersInvoked` is `0` while `projectsCount` is `> 0`, the run is invalid for this day — do **not** write the file; mark the day as failed and proceed.
- Projects skipped (with reasons): malformed output, turn-budget exceeded, etc.
- Wave count and batch size actually used (helpful when diagnosing slow runs).
- **Category review summary** from step B.5: replacements made (`from → to`) and slugs approved as new.
- `newCategories` count (with the slugs themselves if ≤ 10).
- **Git status of the journals directory** if it's a git repo: `clean`, `modified (N files)`, or `not a git repo`. Do not run `git add`/`commit`/`push`. (This per-day check is intentionally repeated once at the end of the run in section C — the per-day reading is a snapshot at that point, the end-of-run reading is the canonical final state. Both are useful; the duplication is by design, not a bug.)
- Whether a companion `<targetDate>.user.md` exists — useful context for the user but do not open or read it.

### C. Cross-day summary

After every enumerated date has been processed, emit a single consolidated summary covering:

- **Per-day list**: one line per `targetDate` with file path (or "not written"), `analyzersInvoked`, `accomplishmentsCount`, and any error reason.
- **Totals**: total days requested, days written, days skipped/failed.
- **Aggregated counts**: total projects covered, total sessions, total accomplishments across all written days.
- **Aggregated category review**: replacements and new slugs across the run.
- **Git status of the journals directory** at the end of the run (single check; not per-day).

For a single-date input, this collapses to one per-day record — keep the format consistent so the user always knows where to look.

## Constraints

- **Never read JSONL content into your own context** — by any means. This includes `Read`, `Bash` (`cat`/`head`/`grep`/`jq`/`python3 -c "...print(...)"`), `Grep`, and any other tool. The only permitted JSONL operation in this skill is the boolean existence check in step B.2 (does any record fall in range, yes or no). All content extraction — message text, accomplishments, file paths, commits, categories, enrichment fields — must happen inside a `journal-analyzer` subagent invocation (`Agent` tool). The analyzer has its own context window and is the only component authorized to author the structured signal the resume-builder consumes. **There is no "save tokens by extracting it myself" shortcut** — bypassing the analyzer drops the inferential enrichment fields (`starComponents`, `qualityTier`, `scope`, `context`, `impact`) and silently breaks the downstream resume contract.
- **"Agent tool errored" is never a license to substitute inline parsing.** If `journal-analyzer` cannot be invoked for any reason, abort that day's run and write nothing for it — do not produce a journal file by parsing JSONL yourself.
- **Do not duplicate or rewrite** the analyzer's accomplishment text. The resume-builder depends on stable, attributable bullets — paraphrasing breaks that.
- **Do not invent themes that aren't supported** by the actual accomplishments. Themes are observations, not aspirations.
- **Frontmatter is the source of truth.** If the body and frontmatter disagree, downstream tools will trust the frontmatter — keep them in sync.
- **Be honest about gaps.** If an analyzer failed for a project, list it in the per-day status (and as a comment in the frontmatter if you like) rather than silently dropping it.
- **Time zones matter.** When the user says "today", they mean their local day, not UTC. Confirm with the system date if unsure.
- **Process ranges serially, not in parallel.** Day N's `knownCategories` includes day 1..N-1's slugs; parallel days defeat the soft-consistency convergence.
- **Never alter git state in the journals repo.** Writing/appending the journal file is fine; `git add`, `commit`, `push`, branch operations, etc. require explicit user consent. The user owns when their journal history gets recorded.
- **Never touch `*.user.md` files.** They are owned by the user and written exclusively by `/career:user-journal`. Do not read, modify, or infer from them.
- **Soft-consistency is not a straitjacket.** If a domain genuinely doesn't exist in `knownCategories`, minting is correct — just expect to see the new slug surfaced in `newCategories` so the user can review.
- **Non-interactive runs** (e.g., when fired by `/schedule`) must not hang on prompts. If the `maxAnalyzers` warn-and-confirm threshold trips and there's no user to confirm, fail that day cleanly so the schedule system logs it — do not auto-proceed.
